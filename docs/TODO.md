11. **Нотификация о новой записи → (позже) своя CRM.** Триггер общий: пост-хук после успешного `book_slot` (создания `Booking`). Делаем в два этапа.

    **11a. Этап 1 — Telegram-уведомление админу/мастерам.** При новой записи бот шлёт сообщение в служебный чат в формате:
    ```
    Новая запись:
    01.02.2026
    Наименование услуги
    Исполнитель
    сумма
    ```
    - Куда слать: chat id в конфиге бизнеса (`config/businesses/<id>/configuration.json`, напр. `notifications.telegramChatId`); отправка через существующий Telegram-клиент.
    - Точка: пост-хук после создания `Booking` (не блокирует ответ клиенту; асинхронно/через очередь желательно, но для MVP можно синхронно после `ok:true`).
    - **Data-зависимости (есть пробелы — закрыть до реализации):**
      - `Исполнитель` (мастер): FSM собирает слот `master`, но `book_slot` его НЕ принимает/НЕ хранит — добавить `master` в `book_slot.parameters` и в модель `Booking` (миграция).
      - `сумма`: FSM её не собирает — вычислять через `lookup_service` по паре услуга×мастер (`priceForMaster`/базовая `price`); хранить на `Booking` (`amount`).
      - `дата` в формате `01.02.2026`: FSM пишет дату свободным текстом («завтра», «суббота») — нужна нормализация даты при записи (иначе в уведомлении будет сырой текст).

    **11b. Этап 2 — интеграция со своей CRM по API.** Тот же пост-хук дополнительно пушит запись в собственную CRM.
    - Конфиг на бизнес: `integrations.crm` (`endpoint`, креды через env-имена как `channel.telegram.tokenEnv`), маппинг полей записи → полей CRM.
    - Надёжность: идемпотентность по `bookingId`, статус синка на `Booking` (`crmSyncStatus`/`crmExternalId`), backoff-ретраи через BullMQ, fail-open.
    - Наблюдаемость: метрика/лог успехов-ошибок, ручной ре-пуш упавших.
    - Открытые вопросы: формат auth своей CRM, нужен ли двусторонний синк (отмены/переносы из CRM обратно).
