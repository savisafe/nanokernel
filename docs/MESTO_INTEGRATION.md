# Интеграция ai-bot ↔ Mesto CRM — сторона бота

> **От:** команда ai-bot · **Кому:** команда Mesto
> **Статус:** модель согласована. Канонический контракт API —
> `mesto-web/docs/api/bot-integration.md`. Этот файл описывает **нашу
> сторону**: что бот потребляет, как меняется диалог, что мы строим.

---

## 1. Итог согласования

Перешли с односторонней async-модели (наш первый ответ) на **двустороннюю,
где Mesto — авторитет расписания.** Причина: цель продукта — бот **не может
записать на закрытую дату** и **консультирует по реальному расписанию** —
недостижима, пока бот не читает доступность.

- ✅ Бот читает свободные слоты у Mesto (`GET /availability`) и предлагает
  клиенту реальные окна.
- ✅ Бот пишет под валидацией Mesto: создать / перенести / отменить.
- ✅ Бот заводит клиента в `clients` Mesto (имя, телефон, telegram_id).
- ✅ **Блокер `starts_at` снят** (см. §2).

---

## 2. Блокер `starts_at` — снят

В v1 мы не могли отдать `starts_at`: нет календаря, время хранилось свободным
текстом («суббота», «после обеда»). В v2 **Mesto отдаёт конкретные слоты** через
`GET /availability` (уже с offset бизнеса). Поток меняется так:

1. Бот узнаёт услугу и мастера → зовёт `GET /availability`.
2. Показывает клиенту реальные окна Mesto.
3. Клиент выбирает → `starts_at` зафиксирован → бот шлёт его в `POST /bookings`.

Парсер свободного текста даты/времени нам больше **не нужен как блокер**: «суббота»
маппится на доступные слоты субботы из ответа Mesto. Tz живёт в Mesto, мы offset
не выдумываем.

## 3. Что бот потребляет (endpoint'ы Mesto)

| Endpoint                                   | Зачем боту                                              |
|--------------------------------------------|----------------------------------------------------------|
| `GET /availability`                        | Консультация по окнам + получить `starts_at` для записи |
| `POST /bookings`                           | Создать запись (синхронно, по ответу ведём диалог)      |
| `PATCH /bookings/:id`                      | Перенос/изменение                                       |
| `POST /bookings/:id/cancel`                | Отмена (в т.ч. при ложном триггере диалога)             |
| `GET /bookings?telegram_id=`               | Найти запись клиента для отмены/переноса                |
| `POST /clients`                            | Завести клиента (в т.ч. без записи)                     |
| `GET /services`, `GET /team`               | Сверка каталога/мастеров                                 |

Полные форматы и коды ошибок — в `mesto-web/docs/api/bot-integration.md`.

## 4. Новый диалоговый поток (sync, авторитет — Mesto)

```
service → master(опц.) → check_availability → клиент выбирает слот
       → name → phone → confirm → POST /bookings
            ├─ 201            → подтверждаем клиенту реальное время
            ├─ 422 OUT_OF_HOURS / BLOCKED → «это время закрыто», переспрос даты/времени
            ├─ 409 SLOT_CONFLICT          → слот заняли, заново check_availability
            └─ 5xx/сеть (после ретраев)   → «подтвердим в течение N минут» + алерт в чат бизнеса
```

Это **переделка диалогового движка**: раньше `book_slot` подтверждал клиенту
локально и слал fire-and-forget уведомление. Теперь подтверждение клиенту
зависит от ответа Mesto. Важный нюанс: при жёстком сбое Mesto мы **не**
обещаем слот жёстко (не мы авторитет) — отвечаем «подтвердим скоро» и алертим
студию.

## 5. Что строим на стороне бота

| Компонент | Файл | Что делаем |
|---|---|---|
| Навык доступности | `src/modules/skills/skills/check-availability.skill.ts` (новый) | Сейчас `check_availability` объявлен в конфиге, но не реализован. Обернуть `GET /availability`; использовать и в консультации («когда свободна Дарья?»), и в FSM. |
| HTTP-клиент Mesto | `src/modules/skills/mesto-client.service.ts` (новый) | Bearer-ключ per-bot, ретраи/таймауты, типы ответов и кодов. |
| Создание записи | `src/modules/skills/skills/book-slot.skill.ts` | Вместо локального `Booking` + fire-and-forget — синхронный `POST /bookings`, разбор `201/422/409/5xx`. Хук `BookingNotifierService.notifyNewBooking()` уже помечен под CRM — вешаемся туда. |
| Отмена/перенос | новые навыки + связать со сниппетом `cancellation` в конфиге | Найти запись (`GET /bookings` или локальный `Booking`), `cancel`/`PATCH`. |
| Локальный `Booking` | `prisma/schema.prisma` | Добавить `mestoAppointmentId`, `mestoClientId`, `syncStatus` (pending/synced/failed), `syncedAt` — трекинг синхронизации (TODO 11b). |
| Конфиг бизнеса | `config/businesses/<id>/configuration.json` | Блок `crm`: `{ "provider": "mesto", "baseUrl": "...", "apiKeyEnv": "..." }`. Ключ per-business подразумевает бизнес. |
| Каталог | `config/businesses/<id>/data/services.json` | Добавить стабильные `id` услуг → шлём `service_external_id`. Mesto матчит по ним. |

Guardrails в конфиге уже запрещают выдумывать свободное время и график мастера —
после `check_availability` бот сможет отвечать **реальными** данными, не нарушая их.

## 6. Что бот отдаёт по полям `POST /bookings`

| Поле | Статус | Комментарий |
|---|---|---|
| `idempotency_key` | ✅ | `"<bot_id>:<booking_id>"`, стабилен на ретраях |
| `service_external_id` + `service_name` | ✅ | Стабильный id из каталога + каноническое имя |
| `starts_at` | ✅ | Конкретный слот из `GET /availability` |
| `duration_minutes` | ✅ | Из каталога по мастеру |
| `amount` | ✅ | Целые тенге; `null` = «уточняется» |
| `currency` | ✅/опц. | Опускаем → Mesto берёт из бизнеса |
| `client.{name,phone,telegram_id}` | ✅ | phone нормализуем в E.164 на нашей стороне |
| `master_name` | ✅ | `null` если «любой» |
| `source.*` | ✅ | `telegram`, `bot_id`, `conversation_id` |

## 7. Открытые вопросы первого ответа — закрыты

| Тема | Решение |
|---|---|
| `starts_at` (блокер) | Снят: `GET /availability` даёт конкретные слоты (§2). |
| Канал отмены/правки | Есть: `POST /bookings/:id/cancel`, `PATCH /bookings/:id`. |
| `amount` единицы/null | Целые мажорные единицы; `null` = «уточняется», `0` = бесплатно. |
| `currency`/`timezone` | Владелец — Mesto; бот может опустить. |
| Каталог услуг | V1: владеет бот; Mesto матчит по `service_external_id`. |
| Tz Казахстана | `Asia/Almaty`, `+05:00` (не `+06:00`). |
